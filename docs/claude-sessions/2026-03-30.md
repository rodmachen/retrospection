### Topic: Todoist API v1 migration

REST v2 (`/rest/v2/`) and Sync v9 (`/sync/v9/`) both returned 410 Gone — fully deprecated as of March 2026. Migrated client to unified **API v1** (`/api/v1/`):

- Base URL: `https://api.todoist.com/api/v1/`
- Cursor-based pagination (`{ results: [...], next_cursor }`)
- Field renames: `checked` (not `is_completed`), `added_at` (not `created_at`), `inbox_project` (not `is_inbox_project`), `section_order` (not `order`)
- Completed tasks available at `GET /api/v1/tasks/completed/by_completion_date?since=&until=` (max 3 months)
- Webhook `event_data` uses v1 field names

Internal normalized types unchanged — mapping happens in the client layer.

### Topic: Seed debugging — FK violation on section_id

Completed tasks from the API referenced sections that had been deleted/archived. Those sections weren't in our `sections` table (only active sections fetched), causing FK violations. Fix: set `section_id: null` for minimal completed-task rows in the seed.

### Topic: Webhook connection and activation

Key finding: webhooks in Todoist API v1 require completing the **OAuth flow** to activate — a personal API token alone is not sufficient. Steps:
1. Create app in App Management Console, add redirect URI, set callback URL
2. Complete OAuth authorize → capture code from redirect → exchange for access_token
3. This activates webhook delivery for your account

Debugging steps taken:
- Endpoint initially returned 404 — deployment was off `main` branch, not feature branch
- Fixed by promoting `feature/backend-phase1` deployment to production in Vercel
- Vercel Framework Preset wasn't set to Next.js — caused "No Output Directory" build failure
- Callback URL in App Management was `retrospection.vercel.app` instead of `retrospection-pink.vercel.app` — corrected and webhooks started firing

### Topic: Recurring task behavior — manual due date change

Manually moving a recurring task's due date fires `item:updated`, not `item:completed`. The system correctly does not record a `task_completions` row for this. Completion is only recorded when `item:completed` fires (user checks off the task). The seed's `inferRecurringCompletions` heuristic (due date in future = completed today) is a one-time seed concern only.
